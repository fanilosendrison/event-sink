# Event Sink — Specification

## 1. Purpose

Event Sink is a library that provides **append-only, atomic JSONL event logging**.
It receives structured events and writes them to a persistent file, one JSON line
per event, without ever reading, modifying, or deleting past events.

It is agent-agnostic: it does not depend on Pi, Antigravity, or any specific
agent harness. It only depends on the Node.js standard library.

---

## 2. Terminology

| Term | Definition |
|------|------------|
| **Sink** | A configured instance that accepts events and writes them to a file. |
| **Event** | A single JSON object written as one line in the output file. |
| **Envelope** | The fixed top-level fields present on every event. |
| **Details** | The variable, domain-specific payload nested under `details`. |
| **Namespace** | An opaque string identifying the source module (e.g. `"path-guard"`, `"secret-scanner"`). |
| **Agent** | An opaque string identifying the agent harness (e.g. `"pi"`, `"antigravity"`). |

---

## 3. Output Format

### 3.1 File

- **Filename**: `events.jsonl`
- **Location**: `<statsDir>/events.jsonl`, where `statsDir` is provided at sink creation.
- **Encoding**: UTF-8.
- **Line separator**: `\n` (LF).
- **No header**, no footer, no BOM.

### 3.2 Event JSON schema

Every event is a single JSON object on one line. No line breaks inside the JSON.

```json
{
  "timestamp":   "<ISO 8601>",
  "eventId":     "<UUID v4>",
  "agent":       "<string>",
  "namespace":   "<string>",
  "eventType":   "<string>",
  "workspace":   "<string>",
  "sessionId":   "<string>",
  "details":     { ... }
}
```

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `timestamp` | yes | `string` | ISO 8601 UTC (e.g. `"2026-07-05T12:00:00.000Z"`). Provided by caller, or auto-generated by the sink (current time) if omitted. |
| `eventId` | yes | `string` | UUID v4, generated by the sink for each `append()` call. Must be unique per event. |
| `agent` | yes | `string` | Identifies the agent harness. Configured once at sink creation. |
| `namespace` | yes | `string` | Identifies the source module. Configured once at sink creation. |
| `eventType` | yes | `string` | Identifies the event kind. Provided per `append()` call (e.g. `"path_access"`, `"scan_result"`). |
| `workspace` | yes | `string` | Identifies the workspace/project. Configured at sink creation, overridable per `append()` call. An `undefined` value in overrides falls back to the creation-time value. |
| `sessionId` | yes | `string` | Identifies the session. Configured at sink creation, overridable per `append()` call. An `undefined` value in overrides falls back to the creation-time value. |
| `details` | yes | `object` | Domain-specific payload. Provided per `append()` call. Must be a plain JSON object (no functions, no circular refs). |

## 4. API Contract

### 4.1 Factory

```
createEventSink(options) → EventSink
```

#### Options

| Option | Required | Type | Default | Description |
|--------|----------|------|---------|-------------|
| `statsDir` | yes | `string` | — | Directory where `events.jsonl` will be written. Created recursively if missing. Resolved to an absolute path before use. |
| `agent` | yes | `string` | — | Value for the `agent` envelope field on all events. |
| `namespace` | yes | `string` | — | Value for the `namespace` envelope field on all events. |
| `sessionId` | no | `string` | — | Default value for `sessionId`. If omitted, the caller MUST provide it per `append()` call. |
| `workspace` | no | `string` | — | Default value for `workspace`. If omitted, the caller MUST provide it per `append()` call. |

#### Behavior

- `statsDir` is resolved to an absolute path (e.g. `path.resolve`) before directory creation and `filePath` construction.
- If `statsDir` does not exist, it is created recursively (`mkdir -p` semantics).
- No file is created until the first `append()` call.
- Returns an `EventSink` object.

### 4.2 EventSink

#### Properties

| Property | Type | Description |
|----------|------|-------------|
| `filePath` | `string` | Absolute path to the `events.jsonl` file. Read-only. |

#### Methods

```
append(eventType, details, overrides?) → void
```

| Parameter | Required | Type | Description |
|-----------|----------|------|-------------|
| `eventType` | yes | `string` | Value for the `eventType` envelope field. |
| `details` | yes | `object` | Value for the `details` envelope field. Must be JSON-serializable. |
| `overrides` | no | `object` | Per-event overrides (see below). |

**Overrides:**

| Field | Type | Description |
|-------|------|-------------|
| `overrides.timestamp` | `string` | Override the `timestamp` field. If omitted, the sink auto-generates one (`new Date().toISOString()` at append time). |
| `overrides.sessionId` | `string` | Override `sessionId` for this event. An `undefined` value falls back to the creation-time `sessionId`. If neither is provided, the field is written as `undefined`. |
| `overrides.workspace` | `string` | Override `workspace` for this event. An `undefined` value falls back to the creation-time `workspace`. If neither is provided, the field is written as `undefined`. |

#### Behavior

1. The event envelope is assembled: `{ timestamp, eventId, agent, namespace, eventType, workspace, sessionId, details }`.
2. `eventId` is generated as a UUID v4 by the sink.
3. The envelope is serialized as a single-line JSON string.
4. The line is atomically appended to `events.jsonl` (see §5).
5. Returns nothing. Never throws to the caller (see §6.1).

### 4.3 TypeScript Types

```ts
interface EventSinkOptions {
  statsDir:   string;   // required — directory for events.jsonl, created recursively
  agent:      string;   // required — agent identifier ("pi", "antigravity", ...)
  namespace:  string;   // required — source module ("path-guard", "secret-scanner", ...)
  sessionId?: string;   // optional — default sessionId, may be overridden per append()
  workspace?: string;   // optional — default workspace, may be overridden per append()
}

interface AppendOverrides {
  timestamp?: string;   // ISO 8601 — auto-generated by sink if omitted
  sessionId?: string;   // overrides EventSinkOptions.sessionId (undefined falls back to creation-time)
  workspace?: string;   // overrides EventSinkOptions.workspace (undefined falls back to creation-time)
}

interface EventSink {
  readonly filePath: string;
  append(
    eventType: string,
    details: Record<string, unknown>,
    overrides?: AppendOverrides,
  ): void;
}

interface AppendEvent {
  timestamp:  string;
  eventId:    string;   // UUID v4, generated by the sink
  agent:      string;   // from EventSinkOptions.agent
  namespace:  string;   // from EventSinkOptions.namespace
  eventType:  string;   // from append() argument
  workspace:  string;   // from overrides.workspace ?? options.workspace
  sessionId:  string;   // from overrides.sessionId ?? options.sessionId
  details:    Record<string, unknown>;  // from append() argument
}

function createEventSink(options: EventSinkOptions): EventSink;
```

## 5. Atomicity Guarantee

Each `append()` call uses a **temp-file + rename** strategy:

1. Read the current content of `events.jsonl` (empty string if it does not exist).
2. Concatenate: `<current content> + <new JSON line> + \n`.
3. Write the combined content to `<events.jsonl>.tmp.<pid>.<random>` (a temporary file in the same directory with PID and a random component to avoid collisions between concurrent writes from the same process).
4. Atomically rename the temp file to `events.jsonl`.

This guarantees that:
- Every `append()` either fully succeeds or leaves the file unchanged.
- No partial line is ever visible to a reader.
- Multiple processes writing to the same `events.jsonl` see whole lines (though they may miss the most recent writes due to rename timing — this is acceptable).

---

## 6. Error Handling

### 6.1 Contract with caller

- `append()` MUST NOT throw. All errors are caught internally.
- On failure, an error message is written to `process.stderr` with the prefix `[event-sink]`.
- The format is: `[event-sink] Error writing event: <message>\n`.

### 6.2 Failure modes

| Scenario | Behavior |
|----------|----------|
| `statsDir` is a file, not a directory | `mkdirSync` fails silently on construction; first `append()` fails, logs to stderr, event is lost. |
| Disk full | Temp file write fails. Log to stderr. File unchanged. |
| Permission denied | Temp file write or rename fails. Log to stderr. File unchanged. |
| Concurrent rename race (different processes) | One process wins, the other's `renameSync` throws `ENOENT`. The event is lost. Log to stderr. |
| Concurrent writes from same process | Each `append()` uses a unique temp file (`<pid>.<random>`). No collision, both writes succeed, order is non-deterministic. |
| `details` contains non-serializable values | `JSON.stringify` may throw or produce invalid output. Behavior is **undefined** — callers are responsible for passing valid data. |

---

## 7. Non-Goals

Event Sink does NOT:

- **Rotate** files. One file per sink, grows indefinitely.
- **Read** events. It is write-only from the API perspective.
- **Query**, **filter**, or **aggregate** events.

- **Validate** the schema of `details`. Callers own their payload.
- **Enforce** event type taxonomy. `eventType` is an opaque string.
- **Batch** writes. Each `append()` is a full atomic write cycle.
- **Buffer** in RAM. Every append flushes to disk.

## 8. Example Usage (illustrative, not prescriptive)

### Pattern A — sessionId/workspace at creation (most extensions)

```ts
import { createEventSink } from "event-sink";

const sink = createEventSink({
  statsDir: "/var/stats/session-abc",
  agent: "pi",
  namespace: "path-guard",
  sessionId: "abc123",
  workspace: "/home/user/project",
});

sink.append("path_access", {
  toolType: "write",
  action: "redirected",
  givenPath: "/dotpi/foo.ts",
}, {
  timestamp: "2026-07-05T12:00:00.000Z",
});
```

### Pattern B — per-event sessionId/workspace (zero-timeout-filter style)

```ts
const sink = createEventSink({
  statsDir: "/var/stats",
  agent: "pi",
  namespace: "zero-timeout-filter",
});

sink.append("timeout_stripped", {
  originalTimeout: 60,
}, {
  timestamp: "2026-07-05T12:00:00.000Z",
  sessionId: "abc123",
  workspace: "/home/user/project",
});
```

### Pattern C — mixed (read-deduplicator style)

```ts
const sink = createEventSink({
  statsDir: "/var/stats/session-abc",
  agent: "pi",
  namespace: "read-deduplicator",
  sessionId: "abc123",   // default for most events
  workspace: "/w",        // default for most events
});

// Uses defaults
sink.append("file_access", { action: "read", path: "/a.ts" }, {
  timestamp: "2026-07-05T12:00:00.000Z",
});

// Per-event override
sink.append("file_access", { action: "blocked", path: "/b.ts" }, {
  timestamp: "2026-07-05T12:01:00.000Z",
  workspace: "/home/user/other-project",
});
```

## 9. Module Architecture

The library is split into three modules, each with a single responsibility.

```
event-sink/
├── specs/
│   └── event-sink.md
│
└── src/
    ├── atomic-writer.ts       ← I/O only (filesystem)
    ├── event-factory.ts       ← pure logic (envelope construction)
    ├── index.ts               ← public API (composes factory + writer)
    │
    └── __tests__/
        ├── atomic-writer.test.ts   ← §10.10
        ├── event-factory.test.ts   ← §10.11
        └── index.test.ts           ← §10.1 – 10.9 (integration)
```

- No `package.json` — zero npm dependencies. Consumers import directly (Bun / Node `--experimental-strip-types`).
- `__tests__/` co-located under `src/`, following Pi extension conventions.
- Integration tests (`index.test.ts`) touch the filesystem (temp dir). Unit tests (`atomic-writer.test.ts`, `event-factory.test.ts`) are isolated: `event-factory` is pure, `atomic-writer` uses `os.tmpdir()`.

### 9.1 `atomic-writer.ts`

**Responsibility**: atomic append of a single line to a file.

- Receives a `filePath` and a `line` (string).
- Reads the current file content, concatenates the new line, writes to a temp file, atomically renames.
- Temp file name: `<filePath>.tmp.<pid>.<random>`.
- Handles errors: logs to stderr, never throws.
- Creates the file if it does not exist.
- Does NOT create directories (caller is responsible).
- Has zero knowledge of events, JSON, or the event sink domain.

**Exports**:

```ts
function atomicAppend(filePath: string, line: string): void;
```

**Reusable**: yes, outside event-sink (e.g. any JSONL logging, audit trails, CI logs).

### 9.2 `event-factory.ts`

**Responsibility**: build a fully-formed event object from inputs.

- Pure function: no I/O, no side effects, no `Date.now()` except for timestamp fallback.
- Receives creation-time configuration (`agent`, `namespace`, default `sessionId`, default `workspace`) and per-call parameters (`eventType`, `details`, `overrides`).
- Resolves precedence:
  - `overrides.workspace ?? defaults.workspace`
  - `overrides.sessionId ?? defaults.sessionId`
  - `overrides.timestamp ?? new Date().toISOString()`
- Generates `eventId` as UUID v4.
- Returns a plain `AppendEvent` object — no serialization.

**Exports**:

```ts
interface BuildEventInputs {
  agent:      string;
  namespace:  string;
  eventType:  string;
  details:    Record<string, unknown>;
  defaults:   { sessionId?: string; workspace?: string };
  overrides?: { timestamp?: string; sessionId?: string; workspace?: string };
}

function buildEvent(inputs: BuildEventInputs): AppendEvent;
```

**Testable**: yes, without filesystem. All envelope logic unit-testable in isolation.

### 9.3 `index.ts`

**Responsibility**: public API. Composes `atomic-writer.ts` and `event-factory.ts`.

- `createEventSink(options)` factory:
  - Resolves `statsDir` to absolute, creates directory recursively.
  - Captures `agent`, `namespace`, and defaults in a closure.
  - Returns `{ filePath, append(eventType, details, overrides?) }`.
- `append()`:
  1. Calls `buildEvent()` to construct the envelope.
  2. Serializes to single-line JSON.
  3. Calls `atomicAppend()` to write.
- Error handling: wraps `atomicAppend` in try/catch, logs to stderr on failure.

**Exports (public)**:

```ts
function createEventSink(options: EventSinkOptions): EventSink;

// Types
interface EventSinkOptions { ... }
interface AppendOverrides  { ... }
interface EventSink        { ... }
interface AppendEvent      { ... }
```

**Internal-only exports** (used by tests, not part of public API contract):

```ts
// Re-exported for testing convenience only
import { atomicAppend } from "./atomic-writer";
import { buildEvent }   from "./event-factory";
```

### 9.4 Dependency graph

```
index.ts
  ├── import { atomicAppend } from "./atomic-writer"
  └── import { buildEvent }   from "./event-factory"

event-factory.ts
  └── zero dependencies

atomic-writer.ts
  └── node:fs, node:path
```

No circular dependencies. `event-factory` and `atomic-writer` are independent of each other.

## 10. Test Contract

A compliant implementation must pass the following behavioral tests:

Tests are organized by module. Integration tests (via `createEventSink`) are marked **[int]**.

### 10.1 File creation **[int]**
- `append()` creates `events.jsonl` on first call.
- `createEventSink()` with a non-existent `statsDir` creates the directory tree, but not the file.

### 10.2 Event envelope **[int]**
- Every written line contains all required envelope fields: `timestamp`, `eventId`, `agent`, `namespace`, `eventType`, `workspace`, `sessionId`, `details`.
- `eventId` is a valid UUID v4 and unique across events in the same file.
- `agent` and `namespace` match the values passed at creation.
- `eventType` matches the value passed to `append()`.
- `details` preserve the exact structure passed by the caller (JSON round-trip).

### 10.3 Overrides **[int]**
- A per-event `overrides.workspace` takes precedence over the creation-time `workspace`.
- An `undefined` `overrides.workspace` falls back to the creation-time `workspace`.
- A per-event `overrides.sessionId` takes precedence over the creation-time `sessionId`.
- An `undefined` `overrides.sessionId` falls back to the creation-time `sessionId`.
- A per-event `overrides.timestamp` is used as-is.
- An omitted or `undefined` `overrides.timestamp` causes the sink to generate one.

### 10.4 Atomicity & ordering **[int]**
- Multiple sequential `append()` calls result in lines in the same order.
- 50 rapid concurrent `append()` calls all persist without data loss or corruption (each line is valid JSON, total count equals calls made).

### 10.5 Error resilience **[int]**
- `append()` never throws when `statsDir` is unwritable (logs to stderr instead).
- After an error, subsequent `append()` calls to a valid path work normally.
- `append()` never throws when the target file exists and is readable.

### 10.6 Idempotency of creation **[int]**
- Calling `createEventSink()` twice with the same `statsDir` does not corrupt the file. Each sink instance shares the same file.

### 10.7 Timestamp auto-generation
- When `overrides.timestamp` is omitted or `undefined`, the sink generates an ISO 8601 timestamp at append time.
- The generated timestamp is within 1 second of the actual append time.

### 10.8 Absolute path **[int]**
- `filePath` is always an absolute path, even when `statsDir` is provided as a relative path.

### 10.9 Same-process concurrency **[int]**
- Two `append()` calls from different sink instances (same process, same `events.jsonl`) both succeed.
- No event is silently lost due to temp file name collision within the same process.

### 10.10 `atomic-writer` unit tests (no filesystem event-sink)
- `atomicAppend(filePath, line)` creates the file on first call.
- Appending two lines to the same file preserves both in order.
- `atomicAppend` never throws when the target directory does not exist (logs to stderr).
- `atomicAppend` recovers: after a failed call (bad path), a subsequent call to a valid path succeeds.
- 100 rapid sequential appends all persist without data loss.

### 10.11 `buildEvent` unit tests (pure logic, no I/O)
- Returns an object with all 8 envelope fields.
- `eventId` is a valid UUID v4 string.
- Two consecutive calls produce different `eventId` values.
- `agent` and `namespace` match the inputs exactly.
- `eventType` matches the input.
- `details` is the same object reference passed as input.
- `overrides.workspace` takes precedence over `defaults.workspace`.
- `undefined` `overrides.workspace` falls back to `defaults.workspace`.
- `overrides.sessionId` takes precedence over `defaults.sessionId`.
- `undefined` `overrides.sessionId` falls back to `defaults.sessionId`.
- `overrides.timestamp` is used as-is when provided.
- When `overrides.timestamp` is omitted or `undefined`, a generated ISO 8601 timestamp is present.
